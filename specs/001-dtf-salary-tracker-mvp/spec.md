# Feature Specification: DTF工资追踪器 MVP

**Feature Branch**: `001-dtf-salary-tracker-mvp`
**Created**: 2025-11-04
**Status**: Draft
**Input**: DTF工资追踪器MVP：拍照识别排班表、快速打卡、工资实时预估

## Clarifications

### Session 2025-11-05

- Q: 休息日(Rest Day)計薪拆分 → A: 完全依照 MOM 拆帳：雇主要求(≤半日→1日薪，>半日→2日薪 + 超時1.5×OT)；員工要求(½/1/1日薪 + 超時OT)；calculateDailyPay需新增initiatedBy、halfDayThreshold參數
- Q: 公眾假期(PH)計薪實作細節 → A: 使用用戶合約約定工時（可配置）：允許用戶設置「正常日工時」（如8h/9h），PH工作≤正常工時→僅額外給1日基本薪；>正常工時→額外1日基本薪+超出部分1.5×OT
- Q: 合規引擎拆分（法規層 vs 公司層） → A: 完全分層架構：法規層固定MOM規則（不可修改），公司層可按店別/職等/合約覆寫任何規則；需設計規則引擎與優先級系統（MOM基線→公司政策→店別規則→個人合約），每條規則記錄來源與條款引用
- Q: REST vs OFF 語義對齊 → A: 區分法定與非法定休息日：數據模型新增isStatutoryRestDay欄位，僅法定Rest Day適用MOM強制規定（½/1/2日薪+超時OT）；非法定休息日按一般OT規則（超時1.5×）；需用戶或系統標記每週法定Rest Day
- Q: MOM Part IV 覆蓋範圍提示 → A: 強制檢測與限制：薪資超過門檻（非工人>$2,600或工人>$4,500）時，禁用OT/Rest Day計算功能，僅允許記錄基本工時；初次設置時檢測並顯示適用狀態，超過門檻者引導至「基本記錄模式」
- Q: 日薪計算方法 → A: ⚠️ 關鍵修正：禁止使用固定除數（如÷26）；必須基於實際工作制度（5天/5.5天大小周/6天/4天）和當月工作日數動態計算；計算公式：日薪 = 月薪 ÷ 當月實際工作日數；需排除公眾假期；每月工作日數不同需動態計算（如11月5天制=20天，6天制=25天）；用於扣款計算（無薪假等）；需在UI中透明展示計算過程

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 拍照导入整月排班 (Priority: P1) 🎯 核心差异化

新加坡鼎泰丰轮班制员工收到主管发的Excel排班表后，希望快速导入到工资追踪器，避免手动输入30天的排班信息。

**Why this priority**: 这是产品的核心差异化功能。如果OCR识别不准确或用户体验差，产品就失去了存在价值。必须优先验证AI识别的可行性。

**Independent Test**: 用户上传一张真实的鼎泰丰排班表图片，系统在10秒内识别出用户姓名、整月排班（30-31天）、每天的状态（工作/休息/加班/年假等），准确率 ≥90%。用户确认后,日历中显示所有识别的排班。

**Acceptance Scenarios**:

1. **Given** 用户首次打开应用并完成基本设置（姓名、月薪、勤工奖），**When** 用户点击"导入排班"并上传主管发的10月排班表照片（Excel截图或WhatsApp图片），**Then** 系统在10秒内显示识别结果：员工姓名、月份、每天的排班状态（如10月1日 10:00上班、10月2日 REST休息、10月5日 OFF/OT休息日加班）

2. **Given** OCR识别完成，**When** 用户查看识别结果确认页面，**Then** 系统显示：识别到的员工姓名、本月统计（工作日X天、REST休息X天、OFF休息X天、年假X天）、关键日期列表、提供"展开查看全部31天"选项、提供"手动修改"和"重新拍照"按钮

3. **Given** 识别结果显示，**When** 用户发现10月15日识别错误（实际是10:00上班，但识别成12:00），**Then** 用户点击"手动修改"，选择10月15日，修改上班时间为10:00，保存后识别结果更新

4. **Given** 识别结果确认无误，**When** 用户点击"确认无误，导入日历"，**Then** 系统保存排班数据，跳转到主界面，日历中显示整月排班（不同颜色标记：工作日🕙、REST🔴、OFF🟠、AL🔵）

5. **Given** 排班表图片模糊或OCR失败，**When** 识别准确率 <70%，**Then** 系统显示错误提示："图片不够清晰，请重新拍照"，并提供"手动输入排班"fallback选项

### User Story 2 - 每日快速打卡记录 (Priority: P1) 🎯 MVP

员工每天下班后，需要快速记录实际的上下班时间，以便系统计算当天的加班费。

**Why this priority**: 工资计算的准确性依赖于打卡数据。这是MVP的核心价值链路：排班导入 → 打卡记录 → 工资计算。

**Independent Test**: 用户在工作日下班后打开应用，看到今天的预排班时间已自动填充（如预计10:00上班），只需输入实际下班时间（如19:30），系统自动计算加班时长（0.5小时）和加班费（$6.96）。

**Acceptance Scenarios**:

1. **Given** 用户已导入10月排班，今天是10月15日（工作日，预计10:00上班），**When** 用户打开"今日打卡"页面，**Then** 系统自动预填：上班时间10:00（来自排班表）、预计下班19:00（标准8小时+1小时休息）、休息时间1小时（默认选中）

2. **Given** 打卡页面已预填，**When** 用户修改下班时间为19:30，**Then** 系统实时计算并显示：总工作时长8.5小时、正常工时8小时、加班时长0.5小时、加班费$6.96（0.5hrs × $9.28时薪 × 1.5倍）

3. **Given** 用户填写完打卡信息，**When** 用户点击"保存打卡记录"，**Then** 系统保存数据，返回主界面，本月工资预估卡片更新（加班费增加$6.96）

4. **Given** 今天是休息日加班（10月5日 OFF/OT），**When** 用户打开"今日打卡"页面，**Then** 系统显示特殊提示："⚠️ 今天是休息日(OFF)，主管要求加班"，并说明："工作>半日→2日薪 + 超时1.5×OT（MOM规定）"，提示用户选择"雇主要求"或"员工要求"

5. **Given** 今天是培训日（10月7日 C），**When** 用户打开"今日打卡"页面，**Then** 系统显示："📚 今天是培训日(C)"，预填培训时间（如09:30-11:00），并说明："培训时间正常计入工资，不计算加班"

### User Story 3 - 实时工资预估 (Priority: P1) 🎯 MVP

员工希望随时查看本月预估工资，了解基本工资、勤工奖、加班费的明细，知道还有多少天发薪。

**Why this priority**: 这是产品的最终价值输出。用户需要看到"这个月能拿多少钱"才会持续使用应用。

**Independent Test**: 用户在主界面看到工资预估卡片，显示本月总工资（如$2,450）、工资构成（基本$1,770 + 勤工奖$100 + 加班费$580）、进度条（已工作18/22天）、预计发薪日（11月7日，还有9天）。点击"查看详细记录"可以看到每天的加班明细。

**Acceptance Scenarios**:

1. **Given** 用户已导入10月排班并记录了18天的打卡，**When** 用户查看主界面工资预估卡片，**Then** 系统显示：预计总工资$2,450.00、基本月薪$1,770.00、勤工奖$100.00（MC 2天，减半）、加班费$580.00、本月进度已工作18/22天（82%）、预计发薪日2025年11月7日（还有9天）

2. **Given** 工资预估卡片显示，**When** 用户点击加班费"$580.00 [详]"，**Then** 系统展开显示加班费明细：工作日加班15小时$208.80、休息日加班2天（16小时）$222.72、公共假期1天（8小时）$148.48

3. **Given** 用户本月有2天MC，**When** 工资预估计算勤工奖，**Then** 系统显示："勤工奖：MC 2天，减半 → $100.00 ⚠️"（原本$200，减半至$100）

4. **Given** 用户已工作32小时加班，**When** 系统检测到接近月度上限（72小时），**Then** 系统在工资卡片下方显示警告："⚠️ 距离月加班上限(72hrs)还剩40小时"

5. **Given** 用户点击"查看详细记录"，**When** 系统显示每天的打卡和工资明细，**Then** 用户可以看到：每天的日期、上下班时间、工作时长、加班时长、加班费、特殊状态日期（REST/OFF/AL）的标记，可以导出为CSV或PDF

### User Story 4 - 灵活日历视图 (Priority: P1) 🎯 MVP

员工需要查看整月排班，快速找到特定日期的状态（工作/休息/加班/年假），以便调班或查看原始排班表。

**Why this priority**: 日历是用户查看排班的主要入口，必须支持复杂的轮班制（非固定AABB大小周）。这是验证OCR识别结果的可视化界面。

**Independent Test**: 用户在日历视图中看到整月排班，不同状态用不同颜色/图标标记（工作🕙、REST🔴、OFF🟠、AL🔵、支援🟢）。点击任意日期可查看详情（如10月5日 OFF/OT 休息日加班，预计10:00上班）。日历顶部有"查看排班表"按钮，点击可全屏查看原始排班表图片（支持缩放）。

**Acceptance Scenarios**:

1. **Given** 用户已导入10月排班，**When** 用户打开主界面查看日历，**Then** 系统显示：2025年10月日历（Sun-Sat格式）、每天显示图标和时间（如"🕙 10"表示10:00上班）、不同状态用颜色区分（工作日🕙蓝色、REST🔴红色、OFF🟠橙色、AL🔵浅蓝色、支援🟢绿色、培训📚黄色）

2. **Given** 日历显示整月排班，**When** 用户点击10月5日（OFF/OT 休息日加班），**Then** 系统弹出日期详情卡片：日期10月5日（星期日）、状态休息日加班(OFF/OT)、预计上班10:00、备注"休息日加班：基础报酬(½/1/2日薪视工时) + 超时1.5×OT"、提供"修改排班"和"记录打卡"按钮

3. **Given** 用户在日历顶部看到"📸 查看排班表"按钮，**When** 用户点击该按钮，**Then** 系统全屏显示原始排班表图片（支持双指缩放、拖动）、提供"重新导入排班"和"下载图片"按钮、用户可以查看其他同事的排班（方便调班）

4. **Given** 日历显示当月排班，**When** 用户左右滑动切换月份（或点击◀ 2025 ▶），**Then** 系统切换到其他月份日历（如果该月未导入排班，显示"未导入排班，点击导入"）

5. **Given** 用户长按某个日期（如10月10日），**When** 系统弹出快捷菜单，**Then** 用户可以选择：修改排班状态、记录打卡、查看历史打卡、复制日期信息

### User Story 5 - MC记录与勤工奖计算 (Priority: P2)

员工需要记录MC（病假）天数，系统自动计算MC对勤工奖的影响（MC ≤1天全额、2-3天减半、≥4天取消）。

**Why this priority**: 勤工奖是工资的重要组成部分（$200），MC记录直接影响工资准确性。虽然重要，但可以在MVP验证核心功能后再完善。

**Independent Test**: 用户添加MC记录（如10月18日 1天 MC，证明编号MC202510180001），系统自动更新勤工奖状态（MC累计2天，勤工奖减半至$100），工资预估卡片实时更新。

**Acceptance Scenarios**:

1. **Given** 用户本月有2天MC，**When** 用户打开"MC记录"页面，**Then** 系统显示：MC天数2天、影响勤工奖减半至$100、MC记录列表（每条显示日期、天数、证明编号、备注）、年度配额进度条已用2/14天（14%）

2. **Given** 用户需要添加新的MC记录，**When** 用户点击"+ 添加MC记录"，填写日期10月22日、天数1天、证明编号MC202510220001、备注肠胃不适，**Then** 系统保存MC记录，更新本月MC统计（3天），勤工奖变为$0（MC ≥4天取消），工资预估卡片减少$100

3. **Given** 用户误添加MC记录，**When** 用户点击某条MC记录的"删除"按钮，**Then** 系统弹出确认对话框："确定删除此MC记录？删除后勤工奖将重新计算"，用户确认后删除，MC统计和勤工奖实时更新

4. **Given** 用户查看年度MC配额，**When** 用户滚动到MC页面底部，**Then** 系统显示：年度Paid MC已用2/14天、进度条（████░░░░░░░░░░░░░░ 14%）、说明"根据新加坡劳工法，每年有14天带薪病假"

### Edge Cases

**排班表识别异常**:
- 图片模糊、光线不足、排班表被遮挡 → 系统提示"图片不够清晰，请重新拍照"，提供手动输入fallback
- 识别到的员工姓名与用户设置的姓名不匹配 → 系统提示"未找到您的姓名，请确认图片是否正确"，允许用户手动选择行
- 识别到的日期不完整（<28天或>31天） → 系统提示"识别到X天，请确认排班表是否完整"
- 单元格内容无法识别（置信度<70%） → 系统标记该日期为"⚠️ 需确认"，用户手动修正

**打卡记录异常**:
- 用户忘记打卡（当天未记录下班时间） → 主界面显示"⚠️ 需要记录今天的下班时间"，提醒用户补充
- 用户修改历史打卡（如昨天的下班时间记错了） → 允许用户修改，但显示"已修改"标记，工资预估自动重新计算
- 下班时间早于上班时间 → 系统提示错误："下班时间不能早于上班时间，请检查"
- 工作时长超过14小时 → 系统提示警告："工作时长异常（超过14小时），请确认是否正确"

**工资计算异常**:
- 月度加班超过72小时上限 → 系统警告："⚠️ 加班时长超过MOM规定上限(72hrs)，超出部分可能无效"
- MC天数超过年度配额（14天） → 系统警告："⚠️ MC天数超过年度配额，超出部分可能无薪"
- 用户基本工资或勤工奖未设置 → 系统提示"请先设置基本工资和勤工奖"，引导用户完成设置

**数据同步异常**:
- 网络断开时查看排班 → 应用离线模式运行，显示已缓存的排班数据，标记"离线模式"
- 网络断开时添加打卡记录 → 数据保存到本地，显示"待同步"标记，网络恢复后自动上传
- 用户在多个设备登录 → 最新的打卡记录覆盖旧数据（显示"最后更新时间"和"设备"）

**特殊排班场景**:
- 派往其他分店支援（如"TCP"分店代码） → 系统识别为"支援日"，按正常工作日计算，允许用户添加备注（如交通津贴$10）
- 公共假期加班（PH） → 系统自动应用额外1日基本薪+超时1.5×OT规则（见FR-022和附录D，符合Constitution v1.1.0）
- 连续多天年假（AL） → 系统统计年假天数，不影响勤工奖

## Requirements *(mandatory)*

### Functional Requirements

**排班表识别 (OCR)**:
- **FR-001**: 系统必须支持用户上传排班表图片（拍照、相册选择、WhatsApp图片）
- **FR-002**: 系统必须在10秒内完成OCR识别
- **FR-003**: 系统必须识别员工姓名、月份、每天的排班状态（work/rest/off/leave/overtime_on_off_day/training/support/unknown）
- **FR-004**: 系统必须解析时间格式（1000 → 10:00, 1200 → 12:00, "1200 2100" → 12:00-21:00兼职时间）
- **FR-005**: 系统必须识别特殊标记（REST/OFF/AL/OFF-OT/支援/C/CO/PH）
- **FR-006**: 系统必须计算识别置信度（每个单元格的confidence值），整体准确率 ≥90% 才算成功
- **FR-007**: 识别失败时（准确率<90%），系统必须提供手动输入排班fallback选项
- **FR-008**: 系统必须允许用户逐条修正识别错误（修改日期、状态、时间）
- **FR-009**: 系统必须保存原始排班表图片到存储（用于"查看排班表"功能）
- **FR-010**: 系统必须显示识别结果确认页面（员工姓名、月份、本月统计、关键日期列表）
- **FR-010a**: 系统必须在导入排班后，智能标记或提示用户标记每週的法定Rest Day（默认识别每週第一个REST/OFF为法定，用户可调整）

**日历视图**:
- **FR-011**: 系统必须显示月度日历视图（Sun-Sat格式）
- **FR-012**: 系统必须用不同颜色/图标标记不同状态（工作🕙、REST🔴、OFF🟠、AL🔵、支援🟢、培训📚）
- **FR-013**: 系统必须支持点击日期查看详情（日期、状态、预计时间、备注）
- **FR-014**: 系统必须支持切换月份（左右滑动或点击◀ ▶按钮）
- **FR-015**: 系统必须提供"查看排班表"按钮，全屏显示原始图片（支持缩放、拖动）
- **FR-016**: 系统必须支持长按日期弹出快捷菜单（修改排班、记录打卡、查看历史）

**打卡记录**:
- **FR-017**: 系统必须在工作日预填上班时间（来自排班表）
- **FR-018**: 系统必须允许用户修改上班时间（如果实际打卡时间不同）
- **FR-019**: 系统必须允许用户输入下班时间
- **FR-020**: 系统必须允许用户选择休息时长（0.5/1/1.5小时，默认1小时）
- **FR-021**: 系统必须实时计算总工作时长、正常工时、加班时长、加班费
- **FR-022**: 系统必须识别特殊日期类型并应用正确的计算规则：
  - 正常工作日：超过用户设置的正常日工时（默认8h）的部分按1.5倍计算OT

## 附錄：關鍵計算公式詳解

### A. 時薪計算（MOM 標準）

**公式**：時薪 = 月薪 ÷ 190.67

**來源**：新加坡MOM官方規定，基於 (52週 × 44小時) ÷ 12個月 = 190.67

**適用範圍**：Part IV 覆蓋的員工（非工人≤$2,600 或 工人≤$4,500）

**示例**：
- 月薪 $1,770 → 時薪 = 1770 ÷ 190.67 = **$9.28**

### B. 日薪計算（⚠️ 關鍵修正）

**❌ 錯誤方式**：日薪 = 月薪 ÷ 26（固定除數）

**✅ 正確方式**：日薪 = 月薪 ÷ 當月實際工作日數

**為什麼不能用固定除數？**
- 不同工作制度的工作日數差異大（5天制 vs 6天制相差5天/月）
- 每月天數不同（28-31天），工作日數也不同
- 使用固定除數會導致誤差高達 20-40%

**工作日數計算規則**：
1. 統計當月符合工作制度的天數
2. 排除公眾假期
3. 考慮工作制度類型

**不同工作制度示例（2025年11月為例，30天）**：

| 工作制度 | 計算規則 | 工作日數 | 日薪（月薪$1,770） | vs ÷26差異 |
|---------|---------|---------|-----------------|----------|
| 5天制 | 週一至週五 | 20天 | $88.50 | +$20.42 (+29.9%) |
| 5.5天制（大小周）| 週一至週五 + 隔週週六 | 22天 | $80.45 | +$12.37 (+18.2%) |
| 6天制 | 週一至週六 | 25天 | $70.80 | +$2.72 (+4.0%) |
| 4天制 | 週一至週四 | 16天 | $110.63 | -$42.55 (-38.5%) |

**誤差影響**：
- **5天制員工**：用÷26會少扣薪水（對員工不利，可能違法）
- **6天制員工**：用÷26會多扣薪水（對雇主不利）

**用途**：日薪用於計算無薪假扣款、休息日基礎報酬等

**實作要點**：
```typescript
// 必須動態計算每月工作日數
function calculateWorkingDays(year: number, month: number, scheduleType: WorkScheduleType): number {
  let workingDays = 0;
  const daysInMonth = new Date(year, month, 0).getDate();

  for (let day = 1; day <= daysInMonth; day++) {
    const date = new Date(year, month - 1, day);
    const dayOfWeek = date.getDay();

    if (isWorkingDay(dayOfWeek, scheduleType)) {
      workingDays++;
    }
  }

  return workingDays;
}

const dailyRate = monthlySalary / calculateWorkingDays(year, month, scheduleType);
```

### C. 休息日計薪（MOM Part IV）

**規則分類**：區分雇主要求 vs 員工要求

#### C.1 雇主要求（Employer-requested）

**基礎報酬**（不含加班）：
- 工作 ≤ 半日（4小時）→ 1日薪
- 工作 > 半日（4小時）→ 2日薪

**加班費**（額外計算）：
- 工作時長 > 正常工時（如8小時）→ 超出部分 × 時薪 × 1.5

**示例**（月薪$1,770，11月5天制，日薪$88.50）：
```
情況A：工作3小時
  基礎報酬：$88.50 (1日薪)
  加班費：$0
  總計：$88.50

情況B：工作8小時
  基礎報酬：$177.00 (2日薪)
  加班費：$0
  總計：$177.00

情況C：工作10小時
  基礎報酬：$177.00 (2日薪)
  加班費：(10-8) × $9.28 × 1.5 = $27.84
  總計：$204.84
```

#### C.2 員工要求（Employee-requested）

**基礎報酬**：
- 工作 ≤ 半日（4小時）→ 0.5日薪
- 半日 < 工作 ≤ 正常工時（8小時）→ 1日薪
- 工作 > 正常工時 → 1日薪 + 超出部分1.5×OT

**示例**（同上）：
```
情況A：工作3小時
  基礎報酬：$44.25 (0.5日薪)
  加班費：$0
  總計：$44.25

情況B：工作8小時
  基礎報酬：$88.50 (1日薪)
  加班費：$0
  總計：$88.50

情況C：工作10小時
  基礎報酬：$88.50 (1日薪)
  加班費：(10-8) × $9.28 × 1.5 = $27.84
  總計：$116.34
```

**MOM參考**：https://www.mom.gov.sg/employment-practices/hours-of-work-overtime-and-rest-days

### D. 公眾假期計薪（⚠️ 修正）

**❌ 錯誤理解**：公眾假期上班整天都是2倍工資

**✅ 正確規則**：
1. 額外給付 1日基本薪（因為月薪已包含假期薪）
2. 超過正常工時部分 × 時薪 × 1.5

**示例**（月薪$1,770，日薪$88.50，時薪$9.28）：
```
工作8小時：
  額外日薪：$88.50
  加班費：$0
  總計：$88.50

工作10小時：
  額外日薪：$88.50
  加班費：(10-8) × $9.28 × 1.5 = $27.84
  總計：$116.34
```

**注意**：月薪制員工的月薪已包含公眾假期的基本報酬，所以上班時只需額外給1日薪，不是2倍

**MOM參考**：https://www.mom.gov.sg/employment-practices/public-holidays-entitlement-and-pay

### E. 正常工作日加班

**規則**：超過正常工時（默認8小時，可配置）的部分 × 時薪 × 1.5

**示例**（時薪$9.28，正常工時8小時）：
```
工作7小時：無加班費
工作8小時：無加班費
工作9小時：(9-8) × $9.28 × 1.5 = $13.92
工作10小時：(10-8) × $9.28 × 1.5 = $27.84
```

### F. 勤工獎計算（公司內規）

**鼎泰豐規則**（示例，實際以合約為準）：
- MC天數 ≤ 1天 → 100% 勤工獎（如$200）
- MC天數 2-3天 → 50% 勤工獎（如$100）
- MC天數 ≥ 4天 → 0% 勤工獎（取消）
