## 项目快速概览
- 技术栈：前端 `React + TypeScript + Vite`；路由 `react-router-dom`；状态 `Zustand`；样式 `Tailwind CSS`
- 数据与后端：`Supabase`（Postgres + RLS + Storage），Serverless API 运行于 `Vercel Edge Functions`
- 测试：`Vitest`（含单元与集成测试，未启用覆盖率报告）
- 构建部署：`Vite` 本地开发与构建，`vercel.json` 管理 Edge 函数；未配置 CI；未提供 Docker
- 文档：`README.md`、`docs/*` 与 `specs/*` 内容较完备（部署与 Supabase 指南、API 参考等）

## 备份方案
- 源代码备份
  - 本地快照：打包整个仓库（含 `.git`、`supabase/`、`api/`、`src/`、`docs/`、`specs/`、配置文件）并加密存储
  - 版本标记：使用语义化标签与变更记录，记录版本号与时间戳
- 数据库与迁移
  - 迁移备份：保留 `supabase/migrations/*` 所有版本及 `supabase/config.toml`
  - 数据备份：导出当前 Postgres 数据快照（通过 Supabase 备份或 `dump`），并校验恢复流程
- 环境与机密
  - 备份 `.env.example`、`vercel.json` 中函数环境允许清单，导出 Vercel 项目环境变量
  - 明确区分前端 `VITE_*` 与后端机密（不含 `SERVICE_ROLE_KEY` 等高权限密钥）
- 对象存储
  - 备份关键桶（如 `schedule-images`），记录访问策略与路径规范
- 异地冗余
  - 至少两处：本地加密盘 + 云对象存储（如 S3/Drive）；记录校验哈希与时间戳

## 项目分析方法
- 架构拓扑
  - 前端 SPA（路由与受保护页面）、服务层封装 Supabase 操作、Edge 函数处理 OCR/薪资计算
- 代码质量评估
  - 可读性与一致性：检查命名、目录结构、类型使用；缺少 ESLint/Prettier 导致规范约束不足
  - 可维护性：模块边界清晰度、公共工具与状态管理合理性
  - 性能指标：构建产物体积、首屏加载、网络请求次数与缓存策略
- 文档完整性
  - 对照 `docs/*` 与 `specs/*`：标注空白或过时内容（如最新 RLS 政策变更是否已反映）
- 测试覆盖率验证
  - 启用 Vitest 覆盖率并生成基线报告；统计语句、分支、函数、行覆盖率

## 改进点识别
- 技术债务
  - 缺失 ESLint/Prettier 配置与统一代码风格
  - 无 CI 流水线（构建、测试、覆盖率与安全扫描未自动化）
  - 未提供容器化方案，难以确保环境一致性（可选项）
  - 路由保护与错误边界可进一步标准化
- 安全
  - 机密注入需持续校验：前端仅使用 `VITE_*`；后端通过 Vercel 环境注入
  - RLS 策略最近有两次调整，需进行生产核验与回归
  - 公共桶访问策略审查（最小权限原则、签名 URL 使用）
  - 边缘函数防护：速率限制、输入校验、审计日志
- 性能
  - 数据库索引与查询模式检查（`employee_id`、`created_at` 等高频查询字段）
  - 客户端缓存与请求合并；减少不必要的往返
  - 构建产物优化：分包、懒加载、Tailwind 树摇
  - 图片与 OCR：压缩策略与并发队列控制
- 用户体验
  - Loading Skeleton、错误提示与重试、网络离线提示
  - i18n 兜底与文案统一、可访问性（语义标签与可聚焦元素）
