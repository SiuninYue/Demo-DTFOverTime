本文档提供3个典型案例，展示ann_ai在不同场景下的优化效果。

## 案例 1：简单营销文案优化（BASIC模式）

### 场景描述
用户需要快速生成一封营销邮件，但输入非常模糊

### 用户输入
```
帮我写个营销邮件
```

### 处理过程

**DECONSTRUCT（解构）**
- 核心意图：生成营销邮件内容
- 关键实体：邮件（类型未知）、营销（目的）
- 上下文：完全缺失
- 输出要求：无
- 约束：无

**DIAGNOSE（诊断）**
- 问题1：缺少产品/服务信息
- 问题2：缺少目标受众
- 问题3：缺少语气和风格要求
- 问题4：缺少输出长度限制
- 复杂度判断：Simple → **BASIC模式**

**DEVELOP（开发）**
- 选择技术：角色分配 + 输出规格 + 结构化
- 角色：营销文案专家
- 结构：开头（钩子）→ 正文（卖点）→ 结尾（CTA）
- 自动判断平台：通用场景 → ChatGPT

**DELIVER（交付）**

### 优化输出

```
**Your Optimized Prompt:**

你是一位资深营销文案专家。请为 [产品/服务名称] 撰写一封营销邮件，目标受众是 [目标群体]。

邮件结构：
- 开头：引人注意的钩子（1-2句话，突出痛点或利益）
- 正文：阐述3个核心卖点和客户利益
- 结尾：明确的行动召唤（CTA），引导下一步操作

要求：
- 语气：[专业/友好/紧迫]（请选择）
- 长度：约200字
- 避免过度营销语言，保持真诚

**What Changed:**
添加了营销专家角色定位，明确了邮件三段式结构，提供了可填充的参数框架（产品、受众、语气），确保输出符合营销邮件标准格式。
```

### 预期效果
- 用户填充参数后即可获得结构完整的营销邮件
- 明确的卖点和CTA提升转化率
- 可复用的框架适用于不同产品

## 案例 2：复杂技术文档需求（DETAIL模式）

### 场景描述
用户需要为开源项目撰写API文档，需求较为复杂

### 用户输入
```
我要写一个REST API的文档，怎么写比较好
```

### 处理过程

**DECONSTRUCT（解构）**
- 核心意图：生成REST API文档
- 关键实体：REST API、文档
- 上下文：开源项目（从场景推断）
- 输出要求：不明确
- 约束：无

**DIAGNOSE（诊断）**
- 问题1：文档类型不明确（用户指南 vs 开发者文档）
- 问题2：缺少API复杂度信息
- 问题3：缺少目标读者画像
- 问题4：未说明格式偏好
- 复杂度判断：Complex（专业领域 + 多步骤）→ **DETAIL模式**

**DEVELOP（开发）**
- 触发DETAIL协议 → 提出针对性问题：
  1. API的主要功能和端点数量？
  2. 目标读者是谁（初学者/有经验的开发者）？
  3. 期望的文档风格（简洁实用 vs 详尽教程）？

**假设用户回答**：
- 功能：用户管理API，包含CRUD操作（约5个端点）
- 读者：有一定经验的后端开发者
- 风格：简洁实用，带代码示例

**继续DEVELOP**
- 选择技术：任务分解 + Few-shot示例 + 结构化框架
- 角色：技术文档工程师
- 结构：概述 → 认证 → 端点详情 → 错误处理 → 示例
- 自动判断平台：长文档 + 结构化 → Claude

**DELIVER（交付）**

### 优化输出

```
**Your Optimized Prompt:**

你是一位经验丰富的技术文档工程师。请为以下REST API创建开发者文档：

<context>
API名称：用户管理API
功能：用户的创建、读取、更新、删除（CRUD）操作
目标读者：有一定经验的后端开发者
</context>

<task>
按以下结构撰写文档：

## 1. API概述
- 简要说明API的用途和核心功能（2-3句话）
- 基础URL和版本信息

## 2. 认证方式
- 说明认证机制（如JWT、API Key）
- 提供认证示例代码

## 3. 端点详情
对每个端点提供：
- HTTP方法和路径
- 请求参数（路径/查询/请求体）
- 响应格式（成功/失败）
- 状态码说明

按以下格式组织：
```
### POST /api/users
创建新用户

**请求体：**
| 字段 | 类型 | 必需 | 说明 |
|------|------|------|------|
| username | string | 是 | 用户名 |
| email | string | 是 | 邮箱地址 |

**响应示例：**
{
  "id": "123",
  "username": "john_doe",
  "email": "john@example.com",
  "created_at": "2025-01-15T10:30:00Z"
}
```

## 4. 错误处理
- 列出常见错误码及含义
- 提供错误响应格式

## 5. 完整使用示例
提供一个实际场景的端到端示例（如：创建用户 → 获取用户信息 → 更新用户）

</task>

<requirements>
- 代码示例使用 curl 和 JavaScript (fetch)
- 响应格式使用JSON
- 保持简洁实用，避免冗余解释
- 确保所有参数和字段都有清晰说明
</requirements>

**Key Improvements:**
• 添加了技术文档工程师角色，确保专业性和准确性
• 使用XML标签组织复杂结构，清晰划分上下文、任务、要求
• 提供了完整的6段式文档框架，涵盖从概述到示例的所有关键部分
• 为端点详情提供了可复用的表格模板，确保格式一致性
• 明确了代码示例语言（curl + JavaScript），避免歧义

**Techniques Applied:**
- Task Decomposition（任务分解）：将文档拆分为6个独立部分
- Few-shot Learning（少样本学习）：提供端点文档格式模板
- Context Layering（上下文分层）：使用XML标签组织信息层级
- Platform Optimization（平台优化）：针对Claude的长文本和结构化能力

**Pro Tip:**
这个提示词生成的文档可以直接作为README.md或docs/API.md使用。如果API端点超过10个，建议进一步将端点分组（如：用户管理、权限管理等）以提升可读性。
```

### 预期效果
- 生成完整的API文档，包含所有必需部分
- 统一的格式确保专业性
- 代码示例降低学习曲线
- 可直接用于开源项目

## 案例 3：跨平台优化对比

### 场景描述
同一个创意写作需求，针对不同AI平台的优化策略

### 用户输入
```
帮我构思一个科幻短篇小说的大纲，主题是时间旅行
```

### 处理过程

**共同分析（DECONSTRUCT + DIAGNOSE）**
- 核心意图：生成科幻小说大纲
- 关键实体：科幻、短篇小说、时间旅行
- 复杂度：Moderate（创意类 + 需要结构）
- 问题：缺少人物设定、冲突点、故事走向

---

### 优化版本 A：针对 Claude

**策略选择**：
- 利用Claude的长上下文能力
- 强调深度推理和多层次冲突
- 使用思维链引导创意发散

**优化输出**：

```
你是一位资深科幻小说编剧，擅长构建复杂的时间旅行叙事。请为以下科幻短篇小说创建详细大纲：

<theme>时间旅行</theme>

<approach>
让我们一步步构建这个故事：

第一步：设定核心悖论
- 思考一个独特的时间旅行机制（避免常见的"回到过去改变历史"套路）
- 设计一个时间悖论作为故事的核心冲突

第二步：角色设计
- 主角：动机、性格缺陷、与时间旅行的关系
- 对手/对立力量：可以是人物、时间本身、或道德困境

第三步：三幕结构
- 第一幕：触发事件，主角如何获得/发现时间旅行能力
- 第二幕：冲突升级，时间旅行带来的意外后果
- 第三幕：高潮与解决，如何处理悖论，结局（开放/封闭）

第四步：主题深化
- 探讨的哲学问题（自由意志 vs 宿命论？记忆与身份？）
- 情感核心（失去与救赎？责任与选择？）
</approach>

<requirements>
- 大纲长度：适合5000-8000字短篇小说
- 包含2-3个情节转折点
- 确保时间旅行的内在逻辑自洽
- 提供开头场景的具体描述（1段）
</requirements>

请按照上述步骤思考，然后提供完整的故事大纲。
```

**特点**：
- 使用XML标签清晰组织
- 思维链（一步步思考）引导深度创意
- 强调逻辑自洽和哲学深度
- 适合Claude的推理能力

---

### 优化版本 B：针对 Gemini

**策略选择**：
- 利用Gemini的创意和多样性
- 鼓励探索性思维
- 提供灵感方向但不限制

**优化输出**：

```
你是一位充满想象力的科幻小说创作者。请为"时间旅行"主题构思一个短篇小说大纲，要求：

**创意方向（选择或混合）：**
1. 硬科幻路线：基于物理学理论（虫洞、相对论、量子力学）
2. 情感驱动路线：时间旅行作为隐喻（弥补遗憾、寻找失去的爱）
3. 黑色幽默路线：时间旅行带来的荒诞后果
4. 反乌托邦路线：时间旅行被商业化/武器化

**大纲应包含：**
- 一句话概念（High Concept）
- 主角简介（姓名、职业、核心冲突）
- 3-5个关键场景描述
- 一个意外转折
- 结局走向

**风格偏好：**
请创造性地融合至少2种上述方向，提供一个独特且引人入胜的故事。避免常见科幻小说套路（如"杀死希特勒"、"祖父悖论"等），寻找新鲜角度。

**灵感参考：**
可以考虑：非线性叙事、多重时间线交织、时间旅行者的孤独、记忆可靠性等元素。
```

**特点**：
- 提供多个创意方向供选择
- 鼓励"创造性融合"
- 明确排除常见套路
- 开放式探索，适合Gemini的创意能力

---

### 优化版本 C：针对 ChatGPT

**策略选择**：
- 使用结构化对话格式
- 明确的分段和要点
- 适合多轮交互完善

**优化输出**：

```
你是一位科幻小说创作顾问。请为我构思一个时间旅行主题的短篇小说大纲。

## 故事核心
请提供：
- **核心概念**：一句话描述故事独特性（"如果______会怎样？"）
- **时间旅行机制**：如何进行时间旅行？（科技装置/天生能力/意外事故）
- **核心冲突**：主角面临的主要问题

## 角色设定
- **主角**：姓名、背景、动机、缺陷
- **配角**（如有）：与主角的关系、在故事中的作用
- **对立力量**：反派/道德困境/时间本身

## 情节架构
请用以下格式组织：

**起始**：
- 主角的日常世界
- 触发事件（如何接触到时间旅行）

**发展**：
- 主角的目标（为什么要时间旅行）
- 第一次尝试的结果
- 意外后果和问题升级

**高潮**：
- 关键选择时刻
- 悖论或道德困境

**结局**：
- 如何解决核心冲突
- 主角的改变
- 留给读者的思考

## 附加元素
- 故事时间跨度：[具体年代/时间段]
- 情感基调：[悬疑/温情/黑暗/幽默]
- 主题深度：探讨什么问题？（如：宿命 vs 自由意志）

请提供完整大纲，适合5000-7000字短篇小说。
```

**特点**：
- 使用Markdown标题和要点，清晰分段
- 填空式结构，便于多轮对话完善
- 平衡了创意和结构
- 适合ChatGPT的对话风格

## 平台对比总结

| 维度 | Claude | Gemini | ChatGPT |
|------|--------|--------|---------|
| **格式** | XML标签 + 思维链 | 创意方向 + 开放探索 | Markdown分段 + 填空式 |
| **优势** | 深度推理、逻辑自洽 | 创意多样性、新鲜角度 | 清晰结构、易于交互 |
| **适用场景** | 复杂叙事、哲学深度 | 创意发散、风格混合 | 标准化流程、快速迭代 |
| **输出风格** | 详尽分析型 | 灵感激发型 | 实用指导型 |

## 常见问题 Q&A

**Q1：什么时候会自动触发DETAIL模式？**
A：当满足以下任一条件时：
- 涉及专业领域（技术、医疗、法律等）
- 需要多步骤流程
- 用户输入极度模糊（缺失3个以上关键信息）
- 高创意性要求（如小说、剧本）

**Q2：如果我想指定使用某个AI平台怎么办？**
A：直接在输入中说明即可，如："用Claude优化这个提示词：..."，系统会遵循你的选择而不是自动判断。

**Q3：优化后的提示词可以直接使用吗？**
A：是的。所有优化后的提示词都是可直接复制使用的。对于包含参数占位符的（如[产品名称]），填充后即可使用。

**Q4：如何判断优化效果是否好？**
A：检查以下5点：
1. 意图是否清晰无歧义
2. 角色定位是否准确
3. 输出规格是否具体
4. 结构是否逻辑清晰
5. 是否包含必要的上下文

**Q5：可以优化其他语言的提示词吗？**
A：可以。4-D方法论适用于任何语言，只需在输入时说明目标语言即可。
